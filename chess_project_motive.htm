Why Chess ?

As you might know already, chess is one of those board games that has been developed a lot in the computer science world, see here for more info Chess Projects on Github. There are 7,478 public repositories matching the topic of Chess.

Answer:
To really understand how the semantics of chess actual work under the hood and to learn how to build such a difficult system from scratch. For example, chess looks very simple when a person is playing it, but what actually happens under the hood is the cool and hard stuff!


Project Contents

As the project is still very early doors, there is not too much to show off just yet. But the aim of our chess application is to have all of the core features that chess has, within our project. Chess contains the following:

  • A board - 8x8 grid with 64 squares
  • Pieces for each team:
    • 2x Rooks
    • 2x Knights
    • 2x Bishops
    • 1x Queen
    • 1x King
    • 8x Pawns
  • Moving Pieces:
    • legal
    • illegal
  • What do I do ?
    • Checkmate: your king is in danger!
    • Stalemate (Drawing): draw the game and not lose!
    • Winning or Lossing!
  • Special things to do...
    • Castling: swap your king with your rook, to keep your king safe!
    • En passant: a special move only pawns can do!
    • Promotion: our pawn can promote to be either: a queen, rook, bishop or a knight!

Challange 1: Making the Chess Board.

Complete

When I first started this project, I assumed building the chess board would be straightforward - but I quickly realised that wasn’t the case. Initially, I tried to write what I thought was a “perfect” solution from the start. However, as many developers recommend, this isn’t usually the best approach. It’s often better to build a simple brute-force version first and then refactor it into something more efficient and maintainable.

My Brute Force Solution:
  • I created a single parent div containing 64 individual div elements.
  • I manually assigned each square a black or white colour.
  • I also manually placed pieces onto each square.
As you can imagine, this approach was extremely inefficient, repetitive and not scalable at all.

My Optimal Solution:
I then refactored the system into a more structured and dynamic approach:
  • Start with an empty board
  • Populate it programmatically
  • Pass in the pieces that should be placed on the board
  • Use row and column values to generate the 8×8 grid
  • The logic is based on the row index (i):
    • If i is 0 or 7, place major pieces (rooks, knights, bishops, queens, kings)
    • If i is 2, 3, 4 or 5, leave empty squares where pieces can move
    • If i is 1 or 6, place pawns

Note: This is my current implementation for generating the chess board. It may need further refactoring in the future, but for now it works as expected and is significantly better than the original hard-coded version.


utils.jsx

				static ROWS = 8;

				static KING = "king";
				static BISHOP2 = "bishop";
				static KNIGHT2 = "knight";
				static ROOK2 = "rook";
				static PAWN = "pawn";
				static freeSpace = "...........";
				
				static letters = "a b c d e f g";
				static nums = "8 7 6 5 4 3 2 1";
				
				static pieces = [this.ROOK, this.KNIGHT, this.BISHOP, this.QUEEN, this.KING, this.BISHOP2, this.KNIGHT2, this.ROOK2];
				static board = [[], [], [], [], [], [], [], []];
			}

			export default Utils;

Board.jsx

			export function initGameBoard(board, pieces, rows, cols) {
				for (var i = 0; i < rows; i++) {
					for (var j = 0; j < cols; j++) {
				
						if (i === 0 || i === 7) {
							board[i][j] = Utils.pieces[j] + " ";
						}
             
						if (i === 2 || i === 3 || i === 4 || i === 5) {
							board[i][j] = Utils.freeSpace + " ";
						}
                
						if (i === 1 || i === 6) {                
							board[i][j] = Utils.PAWN + " ";
						}
				}
			}	   
		return board;
	}
As you can see, this approach is much cleaner and more scalable than hard-coding 64 individual divs.

Challange 2: Moving the Pieces.

Complete

At the moment, I’m working on generating the legal moves for a given piece based on its position on the board. The basic movement logic isn’t too difficult and I’m already able to move pieces programmatically.

For now, the project only includes three piece types:
  • Knights
  • Rooks
  • Bishops

However, there are still several issues and challenges that need to be solved around move validation and board state handling.

Knight Pieces

The knight movement is actually working very well overall, which is a good sign. However, there is still one small issue that needs to be fixed.

For example: if I move a knight from its starting position to a new square, everything behaves correctly at first. Then, if a rook moves into the knight’s original starting square, the system still incorrectly shows that the knight can move back to that square.

This is not correct, since that square is now occupied by the rook and should no longer be a valid move for the knight.

This is currently the smallest issue compared to the other pieces, but it still indicates that there may be a problem with how board state updates or move validation is being handled after pieces are moved.

Rook Pieces

At first, I thought my rook movement logic was working perfectly. However, after spending quite a bit of time debugging and testing different board states, I discovered that the rook logic actually has several problems.

For example: if I move my rook to the right (to the square where the knight originally was), the rook appears to have only one legal move available. Because of this, I initially assumed the rook system was functioning correctly.

What makes this even stranger is that if I remove all of the pawns from the board, the rook logic starts behaving a bit better. Even then, there is still one square on the right side that the rook refuses to move to for some reason.

This is one of those bugs that seems simple at first, but becomes much more confusing the deeper I debug it.

Bishop Pieces

Just like the rooks, I originally thought my bishop movement logic was working correctly. However, after more testing and debugging, I discovered that the bishops behave in a very strange and inconsistent way.

For example: if I move a bishop diagonally, the game is usually able to calculate its legal moves correctly based on its current position. However, problems start appearing once pawns are involved.

Let’s say I move a pawn forward, creating an open diagonal square for the bishop to move into. In theory, the bishop should now be able to move to that newly available square - but for some reason, it cannot.

What makes this even more confusing is that if I completely remove all pawns from the board, the bishop suddenly starts working much better and can move into those squares correctly. However, when pawns are present, the bishop seems to only move in certain diagonal directions instead of both forward and backward diagonals.

This definitely needs to be fixed because, in a real chess game, pawns are almost always present on the board.

As mentioned above, I’m currently working on fixing these issues in the project, which means progress is paused until they’re resolved.



Challenge 3: Everything Currently Runs on the Frontend.

Future Problem

At the moment, the entire project runs on the frontend. This means the chess game can run directly in the browser once the project is deployed, which is great for accessibility and simplicity. However, this also creates a major challenge moving forward.

Right now, all of the game logic is written in JavaScript and while that works for the current state of the project, the future chess engine will be written in Go. The engine will need access to the current game state so it can calculate and generate moves.

At the moment, the engine itself is still completely undeveloped, while the main game logic is already mostly functional aside from a few bugs. Because the frontend and engine will be written in different languages, we need a reliable way for them to communicate with each other.

The plan is for the frontend (JavaScript) to send the current board state to the backend (Go). The engine can then process the position, calculate its move and return either:
  • the updated board state
  • or
  • the move data needed by the frontend to update the board.
This will likely require building an API or communication layer between the frontend and backend systems.

Why Go ?

I chose Go for this project for a few different reasons. First, I genuinely enjoy writing Go code and one of its biggest strengths is how clean and readable the language is. It makes maintaining and understanding large parts of the codebase much easier.

Another major reason is performance and concurrency. In chess programming, speed is extremely important. A chess engine constantly evaluates positions, generates moves and searches through large numbers of possible game states. Go’s built-in concurrency model makes it a strong choice for handling these kinds of tasks efficiently.

For example, a chess board can be represented as a 2D array consisting of rows and columns, meaning many operations require nested loops like this: 

			for i := 0; i < ROWS; i++ { 
				for j := 0; j < COLS; j++ {	
					// process here... 	
				} 	
			}
Running this sequentially works, but it can become slow as the engine grows more complex. With Go, I can take advantage of concurrency features such as Goroutines, Channels and WaitGroups to process calculations in parallel and improve overall performance.

This is one of the main reasons Go felt like the right choice for building the chess engine.

Building Our Own Engine

This will most likely be the most challenging part of the entire project.

Why ?
Chris and I are planning to build our very own chess engine completely from scratch. Whether this turns out to be a brilliant idea or a terrible one, we’re committed to seeing it through.

At the moment, development on the engine has not officially started yet. However, once work begins, we’ll need to spend a significant amount of time researching how chess engines and AI systems are built.

So far, I’ve watched several videos and done some early reading on chess engine development. Some of the concepts and algorithms we’ll likely explore include:

  • Mini-Max Algorithm
  • Alpha-Beta Pruning
  • And much more...


Idea's for the Engine

  1. Our Own Engine, then a ML Engine:
    Our first goal is to build a traditional chess engine from scratch using classic algorithms and search techniques.

    Once we have a working engine, we then want to experiment with a second version powered by Machine Learning (ML). This could involve technologies such as:
    • Neural Networks
    • Recurrent Neural Networks
    • Reinforcement Learning
    • Self-play training

    The interesting part will be comparing the two engines to see:
    • Which performs better
    • How they approach decision-making differently
    • And what advantages/disadvantages each system has

  2. Engine vs Engine:
    Once our engine reaches a stable state, we would also like to submit it to online chess engine testing platforms where it can compete against engines created by other developers.

    This would give us a great way to:
    • Benchmark performance
    • Identify weaknesses
    • And continuously improve the engine over time
